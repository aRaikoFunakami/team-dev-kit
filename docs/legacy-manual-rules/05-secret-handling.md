# 秘密情報取り扱いルール（旧・人間運用版）

> 🕰️ **これは team-dev-kit 導入前のサンプルです。**
> いまはこのルールを **2 つの Hook + gitleaks** が機械的に強制します。人が「コミット前に
> 目視で確認する」必要はなく、危険なら自動でブロックされます。
> 現役の仕組みは [`../secret-scan.md`](../secret-scan.md) を参照。

---

人間運用の頃、秘密情報の漏洩防止は **作業者の注意力とレビュー指摘**に頼っていた。
ここではその「人が守るべきルール」を残す。これが守れなかった事故を仕組みで防ぐために
team-dev-kit は二重ガード（pre-commit hook と PreToolUse hook）を導入した。

## コミット・公開してはいけないもの

- API キー、アクセストークン、シークレット
- パスワード、認証情報、秘密鍵
- 個人情報・顧客情報（メールアドレス、氏名、電話番号等）
- 上記が混ざった設定ファイル・ログ・スクリーンショット

これらは以下のどこにも書かない。

- ソースコード・設定ファイル
- コミットメッセージ（→ [01-commit-convention.md](01-commit-convention.md)）
- Issue / PR の本文（→ [02-github-workflow.md](02-github-workflow.md) / [03-issue-authoring.md](03-issue-authoring.md)）

## 守るべき手順（手作業版）

1. **コミット前に差分を目視する。** `git diff --staged` で秘密情報が混ざっていないか確認する。
2. **Issue / PR を発行する前に本文を読み返す。** 会話やログから貼った内容に鍵が無いか確認する。
3. **混ざっていたら止める。** 該当箇所を削除・マスキング・匿名化してからやり直す。
   一度 push / 公開した秘密情報はキャッシュ・インデックスされうるので、ローテーション（無効化）も行う。

## 例示値の扱い

ドキュメントによく出る例示キー（例 `AKIAIOSFODNN7EXAMPLE`）や連番文字列は、
本物ではないため許容する。誤検知を避けるため、検出ルールの allowlist に例示値を追加してよい
（現役では `.gitleaks.toml` の `[allowlist]`）。
**「ブロックされない＝壊れている」ではない。** 動作確認は本物に近いランダム値で行う。

## なぜ仕組み化したか

人間運用では、

- 急いでいると目視を飛ばす
- 会話ログをそのまま Issue に貼って鍵が混ざる
- レビュアーも見落とす

という事故が構造的に起き続けた。team-dev-kit は同じチェックを
**コミット時（pre-commit hook）** と **Issue/PR 発行時（PreToolUse hook）** の
2 経路で gitleaks に機械実行させ、危険なら処理自体を拒否する。
人がうっかりしても止まる状態を作ることが、このルールを skill 化・hook 化した狙い。

```text
あなたが git commit       → pre-commit hook → gitleaks → 危険なら commit を拒否
Claude が gh issue/pr     → PreToolUse hook → gitleaks → 危険なら発行を拒否
```
