# 設計ドキュメント記述ルール（旧・人間運用版）

> 🕰️ **これは team-dev-kit 導入前のサンプルです。**
> いまはこの文体・構造の契約を `doc-writing` skill が保持し、設計文書を書くときに
> 自動で適用します。現役の挙動は
> [`kit/skills/doc-writing/SKILL.md`](../../kit/skills/doc-writing/SKILL.md) を参照。
>
> このルールは設計ドキュメント向けです。README など利用者向け導入文書は
> 読み手が違うため必ずしも従わない。

---

本リポジトリで設計ドキュメントを書くときの文体・構造の契約を定める。
対象読者は経営層ではなく **実装担当者**であり、文書の目的は **設計レビューに耐えること**である。
迷ったら `AGENTS.md` の方針を優先する。

## 目的

設計レビューで指摘される可能性がある事項だけを書く。根拠のない記述は書かない。
文章を書くこと自体が目的ではなく、レビュアーが判断を検証できる状態を作ることが目的。

## 設計判断の記述

すべての設計判断について、採用案と不採用案を併記する。次を含める。

- 比較対象（どの選択肢と比べたか）
- 選定理由（なぜ採用したか）
- 不採用理由（なぜ他案を捨てたか）
- トレードオフと副作用（採用によって何を失うか、どこに影響が出るか）

副作用とトレードオフのない設計判断は記述として不完全とみなす。
いつ・なぜその決定に至ったかを残す。

数値・設定値・閾値を持つ判断は、その具体値を優先して書く。
「速い」ではなく実測値、「大きい」ではなく上限値を書く。

## メリット・デメリット形式の禁止

「メリット」「デメリット」というテンプレート見出しで機械的に整理しない。
内容そのものを書くなという意味ではなく、比較対象・選定理由・不採用理由・トレードオフを
**文章として**記述する。

## 文体

- 人間のアーキテクトが設計レビュー用に書く文体を模倣する
- 箇条書きより文章を優先する（手順・列挙が本質の箇所のみ箇条書きを許可）
- 各章の長さを均一化しない。重要な判断には行数を割き、自明な箇所は短く済ませる
- 結論を要約しない。「概要」「まとめ」の章を作らない

## 禁止事項

- ベストプラクティスの列挙
- 抽象表現（「柔軟」「高性能」「効率的」などの曖昧語）
- 過度に平易な単語への言い換え
- 「。」「、」の多用
- 「---」の多用

## 図

モジュール図・シーケンス図・ワークフロー図・データフロー図は mermaid 形式で作成する。

## 避けるべき兆候

仕上がった文書が次に該当する場合、上記ルールに違反している。

- 構造が綺麗すぎる
- 全章の密度が均一
- 設計判断の履歴がない
- トレードオフがない
- 具体的数値が少ない
- 要約とまとめが多すぎる
